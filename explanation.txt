========================================================================
                      NAGAR SETU — PROJECT EXPLANATION
========================================================================

Nagar Setu is a Smart Civic Issue Reporting & Resolution Platform that
bridges the communication gap between citizens and municipal departments.
This document explains the technical architecture, models, workflows,
routing mechanisms, and security structures implemented in the project.

------------------------------------------------------------------------
1. HIGH-LEVEL ARCHITECTURE
------------------------------------------------------------------------
The platform is designed around a single-page portal paradigm driven by
a Django + Django REST Framework (DRF) backend and dynamic JavaScript on
the frontend.

- Backend: Handles requests, processes REST API endpoints, performs database
  queries, handles photo uploads, and authenticates user credentials.
- Database: Uses SQLite (local file) storing information on users,
  departments, issues, and upvotes.
- Frontend: Single page interface powered by Leaflet.js, OpenStreetMap,
  and native HTML5/CSS grid and flex components with JS Fetch API calls.

------------------------------------------------------------------------
2. DATABASE MODEL SPECIFICATIONS
------------------------------------------------------------------------
The schema in dockets/models.py includes three main custom entities:

A. Department (Name)
   - Unique name representation (e.g. Roads, Sanitation, Electricity, Water Supply).
   
B. Issue (Title, Description, Category, Status, Coordinates, etc.)
   - Category matches categories: pothole, garbage_overflow, broken_streetlight, water_leakage.
   - Status transitions: submitted -> in_progress -> resolved.
   - Latitude & Longitude coordinate columns.
   - Photo: ImageField storing uploaded media locally under media/issues/.
   - ForeignKeys: Links to Department (for routing) and User (citizen reporter).

C. Upvote (Issue, Citizen)
   - Unique together constraint on ('issue', 'citizen') ensures only 
     one upvote is recorded per citizen per ticket.

------------------------------------------------------------------------
3. WORKFLOWS & PORTALS
------------------------------------------------------------------------
The system divides views and APIs into two separate flows:

A. Citizen Flow (Citizen Counter - "/")
   - Access: Requires login to file dockets or toggle upvotes. Public dashboard
     is viewable in guest mode.
   - Home screen: Displays transparency metrics (resolving rates, resolution
     times, charts) and interactive map with colored status pins.
   - File docket: Input fields for title, category, description, and photo.
     Includes a map click pin-dropper and a "Use Current Location" button.
   - Track dockets: Dashboard displaying progress status timeline.

B. Staff Flow (Staff Counter - "/staff/")
   - Access: Requires superuser credentials (is_superuser=True) to log in.
   - Mailbox Inbox: Categorized list of all submitted tickets. Can filter
     by Department.
   - Status updates: Select dropdown to advance status through stages.
   - Metrics page: Tracks performance and resolution efficiency.

------------------------------------------------------------------------
4. AUTOMATED ROUTING ENGINE
------------------------------------------------------------------------
When a citizen submits a ticket:
1. The backend parses the category field.
2. It executes a mapping function:
   - 'pothole'          => Routed to "Roads" Department
   - 'garbage_overflow'  => Routed to "Sanitation" Department
   - 'broken_streetlight'=> Routed to "Electricity" Department
   - 'water_leakage'     => Routed to "Water Supply" Department
3. Retrieves the matching Department instance (or creates it if missing).
4. Assigns the Department ID to the Issue and saves it.

------------------------------------------------------------------------
5. GEOLOCATION AND MAP INVALIDATION
------------------------------------------------------------------------
- Map Picker: Calls navigator.geolocation.getCurrentPosition() to grab browser
  GPS coordinates, drops a Leaflet marker, and updates coordinate tags.
- Invalidation: Since maps are rendered inside toggled tabs, switching tabs
  could cause Leaflet to display broken gray tiles. A window size invalidation
  call (leafletMap.invalidateSize()) is triggered on tab selection to force
  re-evaluation of dimensions and load tiles perfectly.
